# Описание проекта Banking Service (Handover Document)

Привет! Это документация для продолжения разработки проекта. Мы построили мощный бэкенд на базе микросервисной архитектуры с использованием Go 1.26, gRPC, PostgreSQL и NATS JetStream. 
Ниже описано текущее состояние проекта и задачи, которые тебе необходимо выполнить для успешной сдачи.

## Текущий стек и архитектура бэкенда

Проект состоит из **5 изолированных микросервисов** и **1 API Gateway**, все написаны на Go:
1. **API Gateway** (`:8080`) — HTTP Reverse Proxy. Принимает JSON HTTP запросы, валидирует их и маршрутизирует по gRPC на нужные микросервисы.
2. **Account Service** (`:50051`) — управляет балансами счетов. В нем настроен фоновый воркер (Transactional Outbox), который гарантирует надежную отправку событий создания аккаунтов.
3. **Transaction Service** (`:50052`) — управляет переводами средств с жесткой идемпотентностью.
4. **Notification Service** (без порта, слушает NATS) — отправляет email уведомления асинхронно.
5. **User Service** (`:50053`) — новый сервис для регистрации пользователей и профилей. *Внимание: пока содержит только заглушки (stubs) бизнес-логики.*
6. **Analytics Service** (`:50054`) — новый сервис для лимитов. *Внимание: пока содержит только заглушки (stubs) бизнес-логики.*

**Базы данных и миграции:**
* Мы используем PostgreSQL. `account-service` и `transaction-service` имеют собственные логические базы (`banking_accounts` и `banking_transactions`).
* При старте контейнеров **автоматически выполняются миграции** через библиотеку `golang-migrate/migrate`. Тебе не нужно применять `.sql` файлы руками!
* *Важно про БД для новых сервисов:* В данный момент `user-service` и `analytics-service` не используют базу данных (возвращают захардкоженные моки). Если в рамках твоей задачи потребуется хранить пользователей, тебе нужно будет добавить новые инстансы/схемы в `docker-compose.yaml` (по аналогии с `postgres-tx`) и передать им переменную `DATABASE_URL`.

## Твои задачи (Бэклог)

### 1. Фронтенд: Web UI (Бонус +10%)
Тебе необходимо разработать пользовательский интерфейс (например, на React + TypeScript), который будет взаимодействовать со шлюзом `localhost:8080`.
* **CORS:** В `API Gateway` уже настроена базовая поддержка CORS для всех доменов `["https://*", "http://*"]`. Если возникнут проблемы с preflight `OPTIONS` запросами, проверь настройки `cors.Handler` в `cmd/gateway/main.go`.

### 2. Интеграция SMTP (10% оценки)
В `notification-service` уже написан код для отправки писем через пакет `net/smtp` (паттерн Адаптер: `internal/notification/infrastructure/smtp/adapter.go`). 
* Сейчас он работает в **Dev-режиме** и просто выводит письма в консоль Docker, потому что переменная `SMTP_HOST` равна `dev`.
* **Что нужно сделать:** Пропиши реальные креды в `docker-compose.yaml` для сервиса `notification-service`:
  ```yaml
  environment:
    - SMTP_HOST=smtp.gmail.com
    - SMTP_PORT=587
    - SMTP_USER=твоя_почта@gmail.com
    - SMTP_PASS=твой_app_password
  ```
* Протестируй флоу создания аккаунта и убедись, что письмо реально приходит на почту.

### 3. Тестирование (QA) (10% оценки)
Сейчас тестов нет вообще (0% покрытия).
* **Unit-тесты:** Напиши юнит-тесты для слоев `UseCase` (особенно в `account-service` и `transaction-service`). Используй моки (например, `testify/mock`) для репозиториев.
* **Интеграционные тесты:** Напиши тесты, которые проверяют, что распределенные транзакции (TransferFunds) корректно сохраняют события в таблицу `outbox_events` и что воркер успешно их оттуда забирает.

### 4. Observability / Мониторинг (Бонус +10%)
В инфраструктуре `docker-compose.yaml` уже подняты контейнеры Prometheus, Grafana и Tempo.
* **Баг фикс:** Контейнер `tempo` сейчас циклически перезапускается. Тебе нужно починить его конфигурацию (вероятно, не хватает файла `tempo.yaml`, примонтированного в контейнер).
* **Интеграция в код:** Самое главное — наши Go-сервисы сейчас **глухие**. Тебе нужно добавить OTel/Prometheus Middleware в `gRPC` клиенты/серверы и `HTTP` роутер шлюза, чтобы они начали отгружать метрики и трейсы. 

Удачи в завершении проекта! Фундамент залит, архитектура строгая (Clean Arch) — осталось навести красоту и покрыть тестами.
